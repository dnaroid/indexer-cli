# Отчёт: насколько indexer-cli экономит токены и где может вредить

Дата: 2026-04-08

## Краткий вывод

Индексатор **реально может сильно экономить токены**, но в основном на **тёплом и актуальном индексе**. Для активной работы код-агента по **dirty workspace** он несёт заметный риск: агент может получить дешёвый по токенам, но **устаревший или искажённый контекст**.

Итоговая оценка:

- **Для экономии токенов — полезен**
- **Для надёжной работы на незакоммиченных изменениях — сейчас рискован**

## Что было проверено

Были просмотрены основные части системы:

- `src/cli/commands/search.ts`
- `src/cli/commands/ensure-indexed.ts`
- `src/cli/commands/index.ts`
- `src/engine/indexer.ts`
- `src/engine/searcher.ts`
- `src/engine/git.ts`
- `src/core/config.ts`
- `src/chunking/*.ts`
- `src/storage/*.ts`

Также были выполнены живые замеры на текущем репозитории.

## Практические замеры

### Размер кода без индексации

Грубая оценка всех поддерживаемых исходников в репозитории:

- файлов: **94**
- строк: **16,156**
- байт: **441,470**
- оценка объёма: **~110,368 токенов**

### Размер контекста через semantic search

Два реальных запроса `search --top-k 5` дали:

1. `incremental indexing git diff`
   - **~2,365 токенов**
   - **339 строк**

2. `architecture snapshot dependency graph entry points`
   - **~2,459 токенов**
   - **383 строки**

Это означает, что при удачной выдаче retrieval действительно может уменьшать объём контекста **в десятки раз** по сравнению с чтением всего репозитория.

### Задержки

- **тёплый поиск**: около **1.35–1.41s**
- **холодный / отставший индекс**: во время ручных запусков поиск перед ответом вызывал incremental reindex примерно по **49s**

Важно: код подтверждает, что `search` вызывает `ensureIndexed()` один раз на запуск. Поэтому корректная формулировка такая: **холодный поиск может быть доминирован автоиндексацией**, а не обязательно “один search всегда индексирует дважды”.

## Где индексатор экономит токены

### 1. Retrieval вместо чтения всего репозитория

`SearchEngine` делает embedding запроса, ищет ближайшие чанки и возвращает только top-k результатов, а не весь код. Это и даёт основной выигрыш по токенам.

### 2. Чанкинг

В `indexer.ts` и `chunking/*.ts` код разбивается на семантические части:

- `imports`
- `types`
- `impl`
- `full_file`
- `module_section`

По последнему снапшоту:

- всего чанков: **348**
- средний span чанка: **~50 строк**
- типы:
  - `impl`: **248**
  - `types`: **75**
  - `imports`: **25**

Это уменьшает вероятность тащить в агент сразу большие файлы целиком.

### 3. Incremental indexing

`ensureIndexed()` и `IndexerEngine` умеют не перестраивать всё с нуля, а копировать неизменённые данные и вектора между снапшотами, индексируя только изменённые файлы.

Это экономит не prompt tokens у LLM напрямую, а **локальные embedding-затраты и время подготовки**.

## Где индексатор может вредить

### 1. Самый серьёзный риск: dirty workspace не учитывается как надо

`ensureIndexed()` сравнивает текущий `HEAD` с `snapshot.meta.headCommit`, а `getChangedFiles()` использует `git diff sinceCommit..HEAD`.

Следствие:

- **staged** изменения могут не попасть в индекс
- **unstaged** изменения могут не попасть в индекс
- **untracked** файлы могут не попасть в индекс

Для код-агента это критично: обычно он работает именно с незакоммиченными правками.

### 2. Ранжирование и показанный код могут расходиться

`searcher.ts` выбирает результаты по **stored vectors**, а потом перечитывает **текущий файл с диска** по line range.

Это означает возможный рассинхрон:

- ранжирование основано на старом снапшоте
- показанный фрагмент уже взят из текущего рабочего дерева

То есть агент может получить красивую, короткую и дешёвую выдачу, но логически она будет собрана из разных состояний проекта.

### 3. Auto-indexing ухудшает интерактивность

`search`, `structure` и `architecture` проходят через `ensureIndexed()`. Поэтому retrieval-команда может внезапно превратиться в долгую переиндексацию.

На тёплом индексе это почти не мешает. На холодном — ломает UX.

### 4. Vector-only retrieval без страховок качества

Сейчас в поиске нет:

- rerank
- lexical fallback
- dedupe / diversification
- hybrid search

Следствие:

- один и тот же файл может доминировать в выдаче
- import-heavy и boilerplate-чанки могут занимать слишком много места
- агенту может понадобиться дополнительное дочитывание файлов, что съедает часть выигрыша по токенам

### 5. Есть мёртвые или недоведённые оптимизации

#### `embeddingContextSize`

Параметр есть в `src/core/config.ts`, но в runtime-логике фактически используется `ollamaNumCtx`. То есть `embeddingContextSize` сейчас выглядит как **нерабочий конфиг-переключатель**.

#### `skipImportChunksInVectors`

Опция существует в `IndexerEngine`, но не выведена в CLI и не зашита в config surface. Значит полезная оптимизация есть в коде, но обычный пользователь её не может нормально включить.

### 6. Rename / snapshot edge cases

Есть дополнительные риски корректности:

- при rename старый путь может пережить incremental snapshot
- новый failed snapshot может повлиять на дальнейшее копирование данных
- удалённый файл может остаться в выдаче с `(content unavailable)`

Это не всегда ломает систему, но снижает доверие к индексу как к единственному источнику контекста для агента.

## Общий вердикт

### Когда индексатор полезен

Индексатор хорош, если:

- репозиторий уже проиндексирован
- работа идёт по более-менее стабильной ветке
- агенту нужен быстрый старт по архитектуре или точечный поиск по коду
- retrieval используется как **первичный фильтр**, а не как единственный источник истины

### Когда индексатор опасен

Индексатор опасен, если:

- агент активно работает по незакоммиченным правкам
- важна точная синхронизация с текущим состоянием workspace
- retrieval-выдача используется без дополнительной проверки файла
- интерактивность важнее фоновой автоиндексации

## Что стоит исправить перед реальным использованием код-агентом

Минимально необходимые улучшения:

1. учитывать **dirty/untracked files** в `ensureIndexed`
2. явно маркировать результаты как **stale snapshot** или **current workspace**
3. развести режимы:
   - поиск по snapshot
   - чтение текущего файла
4. вывести `skipImportChunksInVectors` в **CLI/config**
5. добавить хотя бы базовый **rerank / dedupe**
6. либо удалить `embeddingContextSize`, либо действительно подключить его к runtime

## Финальная оценка

Если кратко:

> На стабильной, уже проиндексированной ветке индексатор помогает и реально экономит токены.  
> В live-edit workflow он может давать агенту дешёвый, но устаревший или искажённый контекст.

То есть проект уже решает задачу **token reduction**, но пока не до конца решает задачу **trustworthy context for coding agents**.
