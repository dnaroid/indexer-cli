# Оценка: indexer-cli — экономия или вред для токенов код-агента

## Где экономит

### 1. Замена grep-каскадов при разведке (BIG WIN)

Без индексатора агент при поиске "где реализована авторизация" делает:

- `grep "auth"` → 200 результатов по всем файлам
- `grep "middleware"` → ещё 100 результатов
- `grep "authenticate"` → ещё 50
- Читает 5-10 файлов целиком, чтобы понять контекст

С индексатором: **один запрос** `indexer search "authentication middleware" --json --top-k 5` → 5 целевых чанков с координатами. Экономия: **3-10x меньше токенов** на этапе разведки.

### 2. `structure` вместо чтения дерева файлов

`indexer structure --json --path-prefix src/engine` даёт агенту символы, их виды и exported-статус — без чтения самих файлов. Для понимания "что здесь вообще есть" это **5-20x экономия** vs чтение файлов.

### 3. `architecture` вместо обратного инжиниринга

Dependency graph, entrypoints, file stats за один вызов vs агент, который читает package.json, а потом grep'ает все импорты. **10-50x экономия** на первичном погружении в проект.

## Где НЕ экономит или вредит

### 1. Перезапрос кода после поиска (OVERHEAD)

`SearchEngine.search()` делает векторный поиск → получает `filePath:startLine-endLine` → **снова читает файл с диска** (`readFile`). Это означает, что векторный поиск — это не замена чтению файла, а **дополнительный шаг перед ним**. Если агент в итоге читает тот же файл целиком — вы потратили токены на:

- Вызов `indexer search` (CLI invocation overhead)
- Эмбеддинг запроса (Ollama latency ~200-500ms)
- Обработка результатов

**Итого: нет экономии, если агент не использует line range для targeted read.**

### 2. Fallback-чанки местами слишком грубые для точного поиска

Важно уточнение: это **не общий путь для всех файлов**, а fallback-поведение `AdaptiveChunker`. Реальный пайплайн сначала пытается использовать language-specific chunking и эвристическое разбиение в `IndexerEngine`, и только потом падает в adaptive fallback. Поэтому утверждение “поиск обычно вернёт весь файл” было слишком сильным.

При этом сам fallback действительно грубый: файлы ≤220 строк → **один full_file chunk**, а для больших файлов чанки 180-220 строк с overlap 20 — всё ещё довольно крупные единицы.

При target 280 tokens/chunk (≈1120 символов), но fallback на AdaptiveChunker при 220 строк — один чанк может содержать 220×40 = ~8800 символов = ~2200 токенов. **Несоответствие заявленного и реального размера чанка.**

### 3. Ollama overhead — скрытая стоимость

Каждый `search` запрос:

- Запускает Ollama embedding (200-500ms CPU/GPU)
- Serializes через SQLite + LanceDB
- холодный `search` всё ещё может упираться во внезапную автоиндексацию

Важная поправка: текущие defaults в коде уже **не** `concurrency=1, batchSize=1`, а `indexConcurrency=2`, `indexBatchSize=8`. Проблема интерактивности остаётся, но отчёт в этой части был устаревшим.

### 4. SKILL.md навязывает "mandatory first move"

> "Before grep/glob/find/explore, run at least ONE indexer-cli command"

Это **вредно** в сценариях:

- Агент знает точный путь → grep найдёт быстрее и точнее
- Поиск литеральной строки → semantic search хуже точного match
- Маленький проект → overhead индексатора не окупается
- Срочная задача → ждать Ollama embedding бессмысленно

### 5. Stale index risk

Исторически проблема была реальной, и базовый fix уже есть: `ensureIndexed()` и ручной incremental `index` теперь учитывают не только `git diff sinceCommit..HEAD`, но и изменения из working tree через `git status --porcelain --untracked-files=all`.

Раньше риск был таким:

- Агент изменил файл, но не закоммитил
- Файл изменён вне git
- Индексатор не переиндексировал

Теперь staged / unstaged / untracked изменения попадают в incremental reindex перед `search` / `structure` / `architecture`, но этот слой всё ещё требует аккуратной path-state логики для rename/delete→re-add сценариев. Даже после этого остаётся ещё и узкое окно, если файл изменился **после** auto-index и **до** чтения snippet'а с диска.

### 6. Score = `1 / (1 + distance)` — сомнительная метрика

Евклидово расстояние на 768-мерных векторах без нормализации → L2 distance страдает от "curse of dimensionality". Косинусное сходство было бы значительно лучше для текстовых эмбеддингов. Ранжирование может быть **ненадёжным**, и агент будет получать не самые релевантные результаты.

## Количественная оценка

| Сценарий                                   | Без индексатора             | С индексатором                          | Дельта       |
| ------------------------------------------ | --------------------------- | --------------------------------------- | ------------ |
| Разведка незнакомого проекта (50 файлов)   | ~15K токенов (grep cascade) | ~3K (search + structure)                | **-80%**     |
| Поиск конкретной функции                   | ~2K (grep → read)           | ~1.5K (search → targeted read)          | **-25%**     |
| Точное совпадение строки                   | ~200 (grep)                 | ~800 (embed + search + read)            | **+300%** ⚠️ |
| Маленький файл (<200 строк)                | ~500 (read file)            | ~800 (search returns whole file anyway) | **+60%** ⚠️  |
| Изменённый, но не переиндексированный файл | ~2K (read current)          | ~2K (read stale + fix)                  | **0% или +100%** |

---

## Вердикт

**Индексатор полезен** для двух задач:

1. **Первичное погружение** в незнакомый проект (structure + architecture)
2. **Семантический поиск** когда не знаешь точное имя/строку

**Индексатор вредит** когда:

1. Поиск точного совпадения (grep быстрее и точнее)
2. Файлы маленькие (чанк = весь файл, overhead бессмысленный)
3. Агент механически использует semantic search там, где нужен literal match
4. Файл меняется между auto-index и финальным чтением snippet'а

## Главные проблемы для исправления

1. **Размер fallback-чанка**: ≤220 строк → весь файл — слишком грубо. Нужно aggressive splitting на уровне функций/классов (у вас есть AST!)
2. **Косинусное сходство** вместо L2 distance
3. **SKILL.md**: жёсткий mandatory first move уже убран; важно сохранить условное правило, а не снова сделать его абсолютным
4. **Auto-index UX**: холодный `search/structure/architecture` всё ещё может упираться во время переиндексации
5. **`embeddingContextSize`**: параметр всё ещё выглядит не подключённым к runtime-решениям chunking/search
